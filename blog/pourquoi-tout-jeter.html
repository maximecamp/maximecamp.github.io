Refonte vs Rénovation : Pourquoi je vous déconseillerai (probablement) de tout jeter

Un client m'appelle pour un audit : son application a 5 ou 10 ans, elle tourne souvent en PHP. Le code est devenu difficile à maintenir, les bugs s'accumulent.

La demande du client est alors radicale : "C'est trop vieux, c'est du code spaghetti. On veut tout raser et tout refaire en Node.js et React."

En tant qu'expert React et Node.js, je devrais sauter de joie. C'est l'assurance d'un contrat long, sur une stack moderne. Pourtant, 6 fois sur 10, ma réponse est : "Non. Gardons l'existant."

Voici pourquoi la refonte totale n'est pas toujours la décision la plus rentable pour votre business.

1. L'illusion de la page blanche

Il est tentant de croire qu'en repartant de zéro, on va éviter les erreurs du passé. C'est faux. Le vieux code, aussi "moche" soit-il, contient une valeur inestimable : la connaissance métier. Chaque ligne de code étrange, chaque "if" imbriqué est souvent la cicatrice d'un bug corrigé il y a trois ans ou d'un cas particulier client.

Joel Spolsky (fondateur de Stack Overflow) a dit que la réécriture totale est la pire erreur stratégique qu'une entreprise puisse faire. Pourquoi ? Parce que pendant que vous passez 12 mois à reconstruire ce qui existait déjà, vous n'innovez plus. Vos concurrents, eux, avancent.

2. Je parle Legacy (notamment PHP)

Beaucoup de développeurs modernes méprisent le PHP. Pas moi. Lors de ma mission chez My Battery Health, j'ai travaillé sur un environnement complexe mêlant React, Node.js mais aussi du PHP/MySQL historique. Le PHP propulse encore 75% du web mondial. Ce n'est pas une technologie morte, c'est une technologie éprouvée. Ma force est de ne pas être sectaire. Je sais plonger dans votre code PHP existant, comprendre la logique, et le réparer ou l'optimiser sans avoir besoin de tout détruire.

Je traite votre code existant comme un patrimoine à valoriser, pas comme un déchet à évacuer.

3. La rénovation progressive

Plutôt que de dynamiter la maison (Refonte), je propose souvent de rénover pièce par pièce. En architecture logicielle, on appelle ça le pattern "Strangler Fig".

  • On garde le "vieux" cœur PHP qui fonctionne (gestion des commandes, base de données).
  • On développe les nouvelles fonctionnalités en Node.js (microservices) ou React par-dessus.
  • Petit à petit, on remplace les morceaux obsolètes, sans jamais arrêter la production.

C'est ce que permet une approche Fullstack réelle. Je peux maintenir votre back-office en PHP le matin, et développer votre nouvelle API Node.js l'après-midi.

4. Sauver le budget pour ce qui compte

Une réécriture coûte cher et ne rapporte rien à court terme (vos utilisateurs ne voient pas toujours la différence).

Une modernisation coûte 30% moins cher et permet de livrer de la valeur tout de suite. Gardez votre budget pour l'UX, pour l'IA, pour le Mobile. Pas pour réécrire ce qui marche déjà.

Conclusion

Vous avez une application vieillissante ? Ne sortez pas le bulldozer tout de suite.

Appelez-moi pour un audit. Je vous dirai honnêtement s'il faut démolir ou si un expert capable de comprendre le passé pour construire le futur suffit à relancer votre croissance.
